# Directory Organizer（プロ整理ワークフロー）

> **概要**: ユーザーが指定したディレクトリ（フォルダー）を、プロのデータ整理の視点でスキャン→診断→再構成する。ファイル移動後のパス参照の自動更新を含む、安全かつ網羅的な整理ワークフロー。

---

## Phase 1: スキャン（現状把握）

**Goal**: 対象ディレクトリの全体像を把握し、ファイル数のベースラインを確定する。

### Step 1.1: 対象の確認

ユーザーに整理対象ディレクトリを確認する。指定がなければ質問する:

```
🗂️ プロ整理ワークフロー、起動します！
整理するフォルダーを教えてください。

※事前にバックアップをとっておくことを推奨します
```

### Step 1.2: ディレクトリツリーの収集

1. 対象ディレクトリの **全サブフォルダとファイル** を再帰的に取得する
2. ネストが深い場合は各サブフォルダも展開し、**完全なツリーマップ** を作成する
3. ベースラインファイル数をカウントし、記録する:
   ```powershell
   [Console]::OutputEncoding = [System.Text.Encoding]::UTF8; (Get-ChildItem -Recurse -File).Count
   ```

> **重要**: ベースラインファイル数は絶対に記録すること。Phase 4の検証で完全一致を確認するために必要。

---

## Phase 2: 診断（問題分析）

**Goal**: 7つの診断観点でディレクトリ構造の問題点を特定し、改善提案を構築する。

### Step 2.0: プロジェクトルールの読み込み（必須）

整理を始める前に、プロジェクト固有のルールファイルがないか確認する。

`.agent/rules/` ディレクトリ内にフォルダ構造に関するルールファイル（例: `folder-structure.md`）が存在する場合は `view_file` で読み込み、以下のようなルールを把握する:
- トップレベルフォルダの新規作成禁止
- コンテンツ出力先のルール
- フォルダ名の重複チェック義務

**整理プランがこれらのルールに違反しないことを保証すること。**
ルールファイルが存在しない場合は、このステップをスキップしてよい。

### Step 2.1: 7つの診断チェックリスト

以下の観点で現状を分析する:

| # | 診断観点 | チェック内容 |
|---|----------|-------------|
| 1 | **重複フォルダ** | 同じ種類のファイルが複数フォルダに分散していないか |
| 2 | **孤立フォルダ** | ファイルが1～2個しかないフォルダがないか（フォルダとして独立させる意味があるか） |
| 3 | **迷子ファイル** | ルート直下に適切なカテゴリフォルダに入るべきファイルがないか |
| 4 | **命名の不統一** | 英語/日本語混在、命名規則のブレがないか |
| 5 | **カテゴリ粒度の不均衡** | あるフォルダだけ肥大化し、別のフォルダがスカスカでないか |
| 6 | **論理的順序の欠如** | フォルダに番号プレフィックスがなく、並び順が意図不明でないか |
| 7 | **ワークフロー不適合** | プロジェクト固有でないファイルがプロジェクト内にあったり、逆にシステム的なファイルがコンテンツフォルダに混在していないか |

### Step 2.2: パス参照の影響調査（★最重要★）

ファイルを移動する前に、**対象ディレクトリ内のファイルが他から参照されていないか** を必ず調査する:

```powershell
# 対象ディレクトリ名でプロジェクト内を全文検索
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8
Get-ChildItem -Path ".agent" -Recurse -Include "*.md" | Select-String -Pattern "{対象ディレクトリ名}" | ForEach-Object { "$($_.Filename):$($_.LineNumber): $($_.Line)" }
```

加えて `grep_search` ツールで以下を **全て** 検索:
- `.agent/workflows/` 全体 → 移動対象のファイル名やパスの断片
- `.agent/skills/` 全体 → 同上
- `.agent/rules/` 全体 → ルール内のパス参照
- 対象ディレクトリ自身 → 内部の相互参照リンク（Obsidianリンク等）

> **注意**: 影響のあるファイルが見つかった場合、再構成プランの「パス更新リスト」セクションに必ず記載すること。Phase 3.5で更新する。

### Step 2.3: 診断レポートの生成

問題点をテーブル形式でまとめ、ユーザーに提示する再構成プランのベースとする。

---

## Phase 3: 再構成（実行）

**Goal**: 診断結果に基づき、ユーザー承認の上でディレクトリ構造を変更する。

### Step 3.1: 再構成プランの作成

以下の要素を含む再構成プランを作成し、**ユーザーに承認を求める**:

1. **問題点サマリー** — テーブル形式で問題と理由
2. **新フォルダ構造** — Before/After のツリー図（コードブロック）
3. **変更サマリー** — 各フォルダの変更内容テーブル
4. **パス更新リスト** — Phase 2.2で発見した参照パスの更新計画
5. **確認ポイント** — 分類に迷うファイルや判断が必要な事項

> **重要**: ユーザー承認なしに移動を開始してはならない。判断に迷うファイルの分類は一問一答で確認すること。

### Step 3.2: フォルダの作成

承認を得た後、新しいフォルダ構造を作成する。

### Step 3.3: ファイルの移動

以下のルールに従って安全に移動する:

**移動ルール:**
- `Move-Item` のフォールバックとして `Copy-Item` + `Remove-Item` を使用する
- Move-Item が無出力で失敗する場合に備え、各ステップで `$ErrorActionPreference = 'Stop'` と try-catch を使用する
- 日本語パスの `Move-Item` は失敗しやすいので、`Join-Path` でパスを組み立てる
- 1ステップ = 1カテゴリ（1フォルダ分）を移動し、**移動前のファイル数と移動後のファイル数を照合して報告する**（例: `○○ done: 5/5 files`）

**命令テンプレート（1カテゴリ分）:**
```powershell
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8
$ErrorActionPreference = 'Stop'
$base = '{対象ディレクトリの絶対パス}'
try {
    Move-Item -Path (Join-Path $base '{src}') -Destination (Join-Path $base '{dst}') -Force
    'OK'
} catch { "FAIL: $_" }
```

### Step 3.4: 旧フォルダの削除

移動完了後、旧フォルダが空であることを確認してから削除する:

```powershell
$remaining = (Get-ChildItem $path -Recurse -File -ErrorAction SilentlyContinue).Count
if ($remaining -eq 0) { Remove-Item $path -Recurse -Force; "Removed: $name" }
else { "WARNING: $name still has $remaining files!" }
```

> **注意**: `$remaining -gt 0` の場合は絶対に削除しないこと。原因を調査し、ユーザーに報告する。

### Step 3.4.5: 番号プレフィックスの詰め直し（リナンバリング）

番号プレフィックス付きフォルダが歯抜けになった場合（例: `01_` `03_` `04_` → `02_` が空で消えた）、番号を詰め直す。

**衝突回避のため2段階リネーム**を使用する:

```powershell
# Stage 1: 全フォルダを一時名（TMP_XX）にリネーム
Rename-Item (Join-Path $base '03_xxx') (Join-Path $base 'TMP_02') -Force
Rename-Item (Join-Path $base '04_yyy') (Join-Path $base 'TMP_03') -Force
# Stage 2: 一時名から最終名にリネーム
Rename-Item (Join-Path $base 'TMP_02') (Join-Path $base '02_xxx') -Force
Rename-Item (Join-Path $base 'TMP_03') (Join-Path $base '03_yyy') -Force
```

> **注意**: 直接 `03_` → `02_` にリネームすると、既存の `02_` と衝突する。必ず一時名を経由すること。

### Step 3.5: パス参照の更新

Phase 2.2で特定した参照パスを全て更新する:

1. 対象ファイルを `view_file` でパス記載箇所を特定
2. `replace_file_content` または `multi_replace_file_content` でパスを更新
3. 更新が失敗した場合、PowerShell の `String.Replace()` で直接ファイル内容を書き換える

**更新対象の優先順位:**
- `.agent/workflows/*.md` — ワークフロー内のファイルパス参照
- `.agent/skills/*/SKILL.md` — スキル内のファイルパス参照
- `.agent/rules/*.md` — ルール内のファイルパス参照
- プロジェクト内の相互参照リンク

---

## Phase 4: 検証

**Goal**: 移動前後のファイル数が完全に一致すること、パス参照が正しく更新されていることを検証する。

### Step 4.1: ファイル数の一致検証

```powershell
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8
$total = (Get-ChildItem $base -Recurse -File).Count
"TOTAL: $total files (Baseline: {Phase1のベースライン数})"
```

**ベースラインと一致しない場合**:
- 原因を調査する（移動漏れ、削除ミス等）
- ユーザーに報告し、判断を委ねる

### Step 4.2: パス参照の動作確認

Phase 3.5で更新したパスが正しいことを確認する:
1. **旧パスの消滅確認**: 更新対象ファイル内に旧パスが残っていないことを確認
2. **新パスの存在確認**: 参照先ファイルが実際に存在するか検証
3. **目視確認**: 更新したファイルを開き、パスが正しいことを確認

### Step 4.3: 最終構造の出力

完了した構造を一覧で出力する:

```powershell
Get-ChildItem $base -Directory | ForEach-Object {
    $c = (Get-ChildItem $_.FullName -Recurse -File).Count
    "$($_.Name)/ ... $c files"
}
```

---

## Phase 5: 報告

**Goal**: ユーザーに完了報告と最終構造を伝える。

### Step 5.1: 完了報告

以下を含むレポートを送信:

1. **Before/After 構造図** — 変更内容が一目でわかる
2. **ファイル数検証結果** — `{ベースライン}/{最終カウント}` の一致確認
3. **パス更新リスト** — 更新したファイルの一覧
4. **適用した整理原則** — どの診断観点に基づいて変更したか

### Step 5.2: フィードバック対応ループ

ユーザーから追加の整理要望が来た場合:

1. 追加変更を実行（Phase 3 の手順に従う）
2. 必要に応じてリナンバリング（Step 3.4.5）
3. ファイル数を再検証（Phase 4.1）
4. パス参照への影響を再チェック（Phase 4.2）
5. 更新後の構造を再報告

---

## 異常系ハンドリング

| 状況 | 対応 |
|:-----|:-----|
| `Move-Item` が無出力で失敗 | `Copy-Item` + `Remove-Item` のフォールバック、または `Join-Path` でパス構築 |
| ベースラインとファイル数が不一致 | 即座に移動を停止し、原因調査。ユーザーに報告 |
| 旧フォルダにファイルが残っている | 削除せず、残存ファイルを確認してユーザーに報告 |
| パス参照の更新漏れが発覚 | 再スキャンし、漏れたパスを追加更新 |
| ユーザーが分類に同意しない | 再構成プランを修正し、再度承認を求める |

## 整理の5原則（判断基準）

整理の判断に迷った場合、以下の原則を上から順に適用する:

1. **MECE原則** — 漏れなくダブりなく。1つのファイルは1つの場所にだけ存在する
2. **探しに行ける構造** — 「あのファイルはどこ？」と思った時、3回以内のフォルダ移動で辿り着ける
3. **番号プレフィックス** — 論理的順序があるフォルダには `01_` `02_` を付与し、表示順を制御する
4. **孤立の吸収** — 1～2ファイルだけのフォルダは、意味のある上位カテゴリに統合する
5. **命名の統一** — 同一階層では英語/日本語を統一する
